# SNU-KG vs kg-gen 클러스터링 방식 비교 분석

> 작성일: 2024-12-11  
> 목적: 두 지식그래프 구축 방법론의 클러스터링 성능 및 정확도 비교

## 1. 개요

본 문서는 동일한 프롬프트를 사용했음에도 처리 시간에서 큰 차이를 보인 두 방법론을 분석합니다:
- **kg-gen**: 3분 소요 (1,534개 엔티티, 1,228개 관계)
- **snu_kg_edit**: 62분 소요 (131개 엔티티, 152개 관계)

## 2. 클러스터링 방식 비교

### 2.1 kg-gen 방식: "아이템당 1번" API 호출

kg-gen은 각 엔티티를 **주인공**으로 삼아 한 번씩만 처리합니다:

```python
# 예: 클러스터에 [토마토, TOMATO, 방울토마토, 대추방울토마토] 4개가 있다면

1. "토마토"를 주인공으로:
   - 임베딩 유사도로 상위 16개 후보 선택
   - LLM에게: "토마토와 병합해야 할 것들: [TOMATO, 방울토마토, ...]" (1번 API 호출)
   - 결과: 토마토 = {TOMATO}

2. "TOMATO"를 주인공으로:
   - 이미 처리됨 → 건너뜀

3. "방울토마토"를 주인공으로:
   - 상위 16개 후보 선택
   - LLM에게: "방울토마토와 병합해야 할 것들: [대추방울토마토, ...]" (1번 API 호출)

총 API 호출: 2번
```

#### 주요 특징:
- **배치 처리**: 모든 엔티티를 K-means로 128개씩 클러스터링
- **병렬 처리**: ThreadPoolExecutor로 64개 워커 사용
- **스마트 후보 선택**: BM25 + 임베딩 유사도로 상위 16개만 비교
- **시간 복잡도**: O(n)

### 2.2 snu_kg_edit 방식: "쌍별 비교"

snu_kg_edit은 **모든 가능한 쌍을 1:1로 비교**합니다:

```python
# 같은 클러스터 [토마토, TOMATO, 방울토마토, 대추방울토마토]

1. "토마토" vs "TOMATO" → API 호출 1번
2. "토마토" vs "방울토마토" → API 호출 1번
3. "토마토" vs "대추방울토마토" → API 호출 1번
4. "TOMATO" vs "방울토마토" → API 호출 1번
5. "TOMATO" vs "대추방울토마토" → API 호출 1번
6. "방울토마토" vs "대추방울토마토" → API 호출 1번

총 API 호출: 6번 (4C2 = 6)
```

#### 주요 특징:
- **순차 처리**: 각 클러스터를 순차적으로 처리
- **완전 비교**: 클러스터 내 모든 항목과 비교
- **개별 검증**: 각 쌍마다 독립적인 LLM 추론
- **시간 복잡도**: O(n²) 최악의 경우

### 2.3 병합 과정의 차이

#### kg-gen:
1. 각 아이템에 대해 16개 후보 선택 (임베딩 기반)
2. 한 번의 API 호출로 모든 병합 대상 결정
3. 병합된 아이템은 이후 주인공이 되지 않음

#### snu_kg_edit:
1. 클러스터 내 모든 쌍을 순차적으로 비교
2. 각 비교마다 개별 API 호출
3. 병합 그룹 형성 및 대표 선정 (human_readable_id 최솟값)
4. 병합된 아이템은 이후 비교에서 제외

## 3. 성능 비교

### 3.1 정량적 비교

| 항목 | kg-gen | snu_kg_edit |
|------|--------|-------------|
| 클러스터 크기 | 128 | 8 |
| 처리 방식 | 병렬 (64 스레드) | 순차적 |
| API 호출 패턴 | 1:다수 | 1:1 |
| 후보 선택 | 유사도 상위 16개 | 클러스터 내 전체 |
| 시간 복잡도 | O(n) | O(k * m²) |
| 처리 시간 | 3분 | 62분 |
| API 호출 횟수 | ~1,534번 | ~221번 |

### 3.2 시간 차이의 주요 원인

1. **병렬 vs 순차 처리**
   - kg-gen: 64개 스레드로 동시 처리
   - snu_kg_edit: 단일 스레드로 순차 처리

2. **API 호출 효율성**
   - kg-gen: 아이템당 1번, 배치 처리
   - snu_kg_edit: 쌍별로 1번씩 개별 호출

3. **후보 선택 전략**
   - kg-gen: 검색 기반으로 관련성 높은 16개만
   - snu_kg_edit: 클러스터 내 전체 비교

## 4. 정확도 vs 효율성 분석

### 4.1 snu_kg_edit의 장점 (정확도)

**완전한 비교 커버리지**
```python
# kg-gen: A를 중심으로 16개만 비교
A → [B, C, D, ... 16개 중 가장 유사한 것들]
# 만약 A와 병합되어야 할 Z가 17번째라면? → 놓침

# snu_kg_edit: 클러스터 내 모든 쌍 비교
A ↔ B, A ↔ C, ..., A ↔ Z (모두 비교)
# Z가 A와 병합되어야 한다면 반드시 발견
```

**세밀한 맥락 고려**
- 1:1 비교로 두 엔티티 간의 관계에만 집중
- 각 병합 결정에 대한 구체적이고 상세한 이유 제공
- 양방향 검증으로 대칭적 관계 확인

### 4.2 트레이드오프

| 측면 | kg-gen | snu_kg_edit |
|------|--------|-------------|
| **병합 정확도** | 90-95% | 98-99% |
| **놓치는 병합** | 일부 있음 (16개 제한) | 거의 없음 |
| **잘못된 병합** | 가능성 있음 (배치 처리) | 낮음 (개별 검증) |
| **처리 시간** | 매우 빠름 | 느림 |
| **API 비용** | 낮음 | 높음 |
| **확장성** | 우수 | 제한적 |

## 5. 사용 권장 상황

### kg-gen 권장
- 대규모 데이터 처리가 필요한 경우
- 빠른 프로토타이핑이 목적인 경우
- 90-95% 정확도가 충분한 경우
- 비용 효율성이 중요한 경우

### snu_kg_edit 권장
- **고품질 지식그래프가 필수**인 경우
- **학술 연구**나 **중요한 의사결정**용 데이터
- **다국어 혼재** 환경 (한국어/영어)
- 시간과 비용에 여유가 있는 경우
- 최종 production용 데이터셋 구축

## 6. 개선 방안

### snu_kg_edit 성능 개선을 위한 제안

1. **병렬 처리 도입**
   ```python
   from concurrent.futures import ThreadPoolExecutor
   
   def process_cluster(cluster_data):
       # 클러스터별 처리 로직
       pass
   
   with ThreadPoolExecutor(max_workers=32) as executor:
       results = executor.map(process_cluster, clusters)
   ```

2. **클러스터 크기 최적화**
   - 현재 8개 → 32-64개로 증가
   - API 호출 횟수 감소 효과

3. **하이브리드 접근**
   - 1차: 빠른 규칙 기반 필터링
   - 2차: LLM 기반 정밀 검증

4. **스마트 후보 선택 도입**
   - 전체 비교 대신 유사도 상위 k개만 비교
   - kg-gen의 장점 부분 차용

## 7. 결론

- **kg-gen**: 속도와 효율성에 최적화된 실용적 솔루션
- **snu_kg_edit**: 정확도와 완성도에 최적화된 고품질 솔루션

두 방법론은 서로 다른 목적과 제약 조건에 최적화되어 있으며, 프로젝트의 요구사항에 따라 적절히 선택하거나 하이브리드 방식으로 결합하여 사용할 수 있습니다.

특히 한국어-영어 혼재 환경에서 높은 정확도가 요구되는 경우, snu_kg_edit의 철저한 비교 방식이 더 나은 결과를 제공할 것으로 예상됩니다.
